We recommend moving from two enrollment systems to one.
Thatch currently operates two parallel enrollment models — rolling windows for new employers and a separate OE cohort operation every November. We recommend collapsing these into a single CSD-first system that runs identically year-round.
The problem
Two systems, two playbooks
Rolling windows and OE cohorts run on different logic, different CS processes, and different comms. Every November, the team manually rebuilds what should be automated.
Extensions are a symptom, not a solution
The 6% enrollment lift from OE 2026 extensions was real — but required 8 cohorts, exclusion lists, and significant CS coordination. That doesn't scale with 30 new sales reps.
We've been solving the wrong problem
Employers don't want to control enrollment windows. They want to control when coverage begins. The window is an implementation detail Thatch should own.
The recommendation
One system, anchored on coverage start date
Every invite — rolling or OE — gets a 30-day window split by a processing cutoff. Enroll before the cutoff: CSD 1. Enroll after: CSD 2. Same logic year-round, no exceptions.
Extensions become structural, not operational
The two-CSD model embeds the extension concept by design. The Jan 4–8 special extension is the only exception — a targeted policy decision, not an operational workaround.
OE 2027 is a capacity problem, not an architecture problem
With one system, OE is when more invites go out simultaneously — not a different operating mode. One CS playbook. One enrollment flow. One mental model for everyone.
